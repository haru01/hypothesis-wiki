# 反証型 問題インタビュー スクリプト（AIRE-ACT-003・CPF）

[[AIRE-ACT-003]] の**方法**を現場で使える会話スクリプトにしたもの。1インタビュー 45〜60分。
**目的は反証**＝ [[AIRE-H-001]] [[AIRE-H-002]] [[AIRE-H-003]] を「壊しに行く」。支持と反証を同じ重みで探す。

## 使う前に（CPFの鉄則＋反証型の追加ルール）

- **過去の事実を聞く。未来の意向・仮定は聞かない。**「〜だったら？」でなく「直近で〜したのはいつ、何をした？」。
- **ソリューションもこちらの仮説も語らない・誘導しない。**「困りましたよね？」は禁句。
- **反証を能動的に探す**: 各テーマで支持例を聞いたら、**必ず対で「逆に、ちゃんと確かめて/捕まえて進んだ例は？」**を聞く。確証バイアスの自己防衛。
- **切実さは実コストで測る**: 「大変だった」でなく「何に・どれだけ（時間/金/手戻り/信頼）を実際に払ったか」。
- **沈黙を待つ。** 一般論は「直近の具体例は？」「そのとき実際どうした？」で事実に降ろす。

## 対抗仮説の listen-for（[[AIRE-ACT-004]] の揺さぶり監査より）

揺さぶり監査で、3仮説が「認知的降伏シナリオ」という**単一物語の子**で、探索が確証方向に偏っていると指摘された。その解毒として、以下の**対抗仮説を同じ重みで潰しに行く**。各テーマの反証側に**対抗プローブ**（下記◆）を織り込む。**成功基準（事前登録の反証条件）は変えない**——対抗プローブはあくまで listen-for（対抗が実コスト付きで出たら、それ自体が当該仮説の反証シグナル）。

| # | 対抗仮説 | 対抗する仮説 | 聞く場所 |
|---|---|---|---|
| A | 当事者性ゆえに専門外出力を**濃く検証する**（丸投げしない） | [[AIRE-H-001]] | §3 ◆ |
| B | 歪みの主因は承認様式でなく**要件変更/通常の技術的負債**（帰属の別解釈） | [[AIRE-H-002]] | §4 ◆ |
| C | 目利きは AI出力の**評価反復で育つ**（never-skilling でない＝re-skilling） | [[AIRE-H-003]] | §5 ◆ |
| D | この痛みに**対価を払う顧客セグメント**として実在するか（CPFの根っこ） | 分析単位 | §6 ◆ |

## 0. 導入・ラポール（3分）

> 本日はありがとうございます。エージェントを使ったものづくりで**実際に何が起きたか**を、良し悪しの評価でなく事実としてうかがいたいです。うまくいった話も、しくじった話も、そのまま教えてください。（録音可否の確認）

## 1. スクリーニング（5分）— 母集団の確認

- 直近3ヶ月で、2〜3名＋エージェントでプロダクト/機能を作りましたか？ 最後にやったのはいつ？
- チーム構成（人＋どんなエージェント）と、あなたの専門／その中で**専門外だった領域**は？

> 条件に合わなければ無理に続けず対象外として記録（母集団の質を守る）。

## 2. 直近エピソードの再現（12分）— 事実の土台

> 直近で、エージェントを使って作った成果物を1つ、**最初から最後まで**順に教えてください。どこを自分がやり、どこをエージェントに任せ、最後どうなりましたか？

- 専門外の部分は、具体的に何をエージェントに任せましたか？
- 出てきたものを、次へ進める前に**どう扱いましたか**（そのまま承認／自分で確認／誰かに見てもらう）？

## 3. H-001 の反証プローブ（10分）— 専門外の委任と承認の実際

支持と反証を**対で**聞く（[[AIRE-H-001]]）。

- 直近で、専門外のエージェント出力を**そのまま承認して進めた**ことはありますか？ そのとき中身はどこまで分かっていましたか？（＝支持側）
- **逆に**、専門外の出力を**自分で確かめてから／誰かに検証してもらってから**進めた例はありますか？ 何をどう確かめましたか？（＝反証側）
- ◆（対抗A）**2〜3名という小さなチームだからこそ**、専門外でも「最後は自分が背負う」と感じて、**あえて濃く確かめた**ことはありますか？ そのとき何を・どこまで確認しましたか？（＝当事者性ゆえの濃い検証。丸投げの反証）
- 承認するかどうかは、実際には何で決めていますか？（時間・締切・信頼・分かる/分からない）

## 4. H-002 の反証プローブ（12分）— 理解せぬ承認の反復とドリフト・実コスト

支持と反証を対で。**「困った?」でなく実際の場面と払ったコストを**（[[AIRE-H-002]]）。

- 承認を重ねた成果物で、**後から**「おかしい／作り直し」になったことはありますか？ 何が・いつ・どう分かり、**どれだけの手戻り**（時間/金）でしたか？（＝支持側・実コスト）
- その歪みは**顧客に影響**しましたか？（誤動作・期待とのズレ・問い合わせ・信頼低下）具体的に何が起きましたか？（H-002 尾部）
- 歪んだ成果物やログを、**次のエージェント作業の入力に**そのまま渡したことは？ その後どうなりましたか？（AI再流入）
- ◆（対抗B）その「おかしくなった」の**主な原因**は、承認の重ね方でしたか、それとも**要件変更・仕様追加・普通の技術的負債**でしたか？ 承認をきちんとしていても**同じ歪みは起きていた**と思いますか？（＝帰属の切り分け。承認様式への帰属の別解釈）
- **逆に**、理解しないまま承認を続けても**問題が起きなかった／その場で必ず捕まえられた**ケースはありますか？ どう捕まえましたか？（＝反証側）

## 5. H-003 の反証プローブ（8分）— 目利き/オーケストレーションと学び方

支持と反証を対で（[[AIRE-H-003]]）。

- エージェント出力の**良し悪しを見抜けず**、後で痛い目を見たことはありますか？ 何が起きましたか？（＝支持側・実害）
- 何を・誰に・どう任せ、いつ自分に引き戻すか（オーケストレーション）で、失敗した／うまくやった具体例は？
- その「見抜く力・任せ方」は、どこで身につけましたか？（OJT／レビュー文化／研修／独学）**十分だと感じますか、足りないと感じますか？**（＝反証側の手がかり）
- ◆（対抗C）**エージェントを使い込む中で、逆に見抜く力・任せ方が上がった実感**はありますか？ 何を・どう繰り返して身についたと思いますか？（＝評価反復で育つ＝re-skilling。「実行を委ねると育たない」の反証）

## 6. クロージング（5分）

- 今日の話で、一番実害が大きかったのはどれですか？（本人の優先順位）
- ◆（対抗D）その困りごとを減らすために、これまでに**実際にお金や工数を払った**ことはありますか？（レビュー外注・検証ツール・追加の人手・自作の仕組みなど）何に・どれだけ払いましたか？（＝対価を払う顧客セグメントの実在確認。**過去の支払い実績**のみ。「払いますか？」の未来意向は聞かない）
- 同じような作り手を紹介いただけますか？（次の対象者）
- 後から思い出したことがあれば後日でも歓迎です。

## 支持/反証 記録シート（対象者ごと）

証拠の階梯タグ（〈発言〉〈自認〉〈実コスト〉〈行動〉）を添える。〈発言〉だけは支持に数えない。

| 対象者 | H-001 支持/反証（タグ） | H-002 支持/反証（タグ） | H-002尾部 顧客混乱/AI再流入 | H-003 支持/反証（タグ） | 対抗A/B/C/D シグナル（タグ） | 一番の実害（本人優先） |
|---|---|---|---|---|---|---|
| 1 | | | | | | |
| 2 | | | | | | |
| … | | | | | | |

対抗列の書き方: A=当事者性で濃く検証 / B=歪みの主因は要件変更・技術的負債 / C=評価反復で目利きが育った / D=過去に実際に払った対価。**対抗が実コスト付き（〈実コスト〉〈行動〉〈支払い〉）で出たら、対応する仮説（A→H-001, B→H-002, C→H-003）の反証シグナルとして数える。** D は分析単位（顧客セグメント実在性）の材料として別途 `/ingest` に渡す。

## 禁止・注意

- **こちらの仮説を口にしない・示唆しない。** ソリューションも出さない。
- 誘導質問（「〜で困りますよね？」）・未来仮定（「もし〜なら」）を成功基準の根拠にしない。
- 支持例だけ集めない。**各テーマで必ず反証側（逆の例）を1問聞く。**
- 対抗プローブ（◆A〜D）は listen-for。**事前登録した反証条件＝テストカードの成功基準は変えない**（後知恵バイアス防止）。対抗が実コスト付きで出たときに反証側へ数えるだけ。

## 記録のコツ（`/ingest` に効く）

- **事実（発言）は原文引用**、**解釈（こちらの推測）と必ず分ける。**
- 支持・反証を上の記録シートで対象者×仮説の○×に落とし、成功基準（支持:反証の非対称・実コストの有無）を機械的に判定できるようにする。
- 想定と違った発言（驚き・仮説外の発見）は別枠でマークし、発見型 [[AIRE-ACT-002]] の結果と突き合わせる。
