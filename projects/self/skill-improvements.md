# スキル改善バックログ（ドッグフーディングのメタ観察）

self 作り直しシミュレーション（2026-07-18 実施 / new-project→formulate→ingest→decide→view を一巡）で
**実際に遭遇した摩擦・曖昧さ・スキーマの穴**の記録。wiki の仮説/活動レコードとは分離した Schema層への改善候補。

> このファイル自体がアドホックな置き場である点も改善項目（#8）。

各項目: 観察した摩擦 / 該当スキル・ファイル / 改善案 / 優先度

## #1 FPF の正式名称と CPF→次ステージの経路が資料間で揺れる ／ 優先度: 高
- **摩擦**: `stage.md` を FPF に更新する際、FPF の正式名称が分からなかった。`CLAUDE.md` の表では FPF重点＝「課題・自分たち仮説」だが FPF が何の略か明記がない。さらに `playbooks/cpf.md` の移行基準は「次ステージ（FPF）」と書くのに、削除前の旧 self は CPF→**PSF** へ飛んでいた（実践とプレイブックが不一致）。
- **該当**: `CLAUDE.md`（ステージ表）／`playbooks/cpf.md`・`playbooks/fpf.md`／`decide` スキル
- **改善案**: 各ステージの正式名称（CPF/FPF/PSF/SPF/PMF のフルスペル）を `CLAUDE.md` に一覧化。CPF→FPF→PSF… の標準経路と「ステージのスキップ可否」を明文化し、`decide` がスキップ時に警告する。
- 今回の対応: `stage.md` に FPF を暫定表記で記入（要確認）。CPF→FPF の標準経路に従った。

## #2 「検証したが確信度が上がらなかった」状態を確信度↔ステータスの整合ルールが表現できない ／ 優先度: 高
- **摩擦**: [[SELF-H-005]] [[SELF-H-007]] は問題インタビューで検討したが、自認はあっても実コストの証拠が薄く、規律に従い確信度を上げず「確信度4・検証中」に据え置いた。しかし `CLAUDE.md` の確信度表では 4＝「二次情報・状況証拠」、5-6＝「検証中で手応え」。**確信度4＋ステータス検証中は数値上ズレて見え**、`lint` の「確信度-ステータス不整合」で誤検出される恐れがある。
- **該当**: `lint` スキル（確信度-ステータス整合チェック）／`CLAUDE.md`（確信度表）
- **改善案**: 「検証したが確信度が上がらなかった（検証中だが手応えなし）」を正当な状態として定義。lint は "検証中かつ低確信度" を不整合でなく「判断保留／実コスト証拠待ち」として扱う。ステータスに `判断保留` を足す案も検討。

## #3 架空・シミュレーションデータの確信度上限（≤8）が規約化されていない ／ 優先度: 高
- **摩擦**: 架空データで検証済みにする際「9=反証しても崩れない/10=観測事実 には上げず上限8」という規律を、各仮説・ACT・ビューに**手で毎回書いた**。`ingest` の運用注記にはあるが `CLAUDE.md` の確信度表・テンプレには無く、守り忘れやすい。
- **該当**: `CLAUDE.md`（確信度表）／`formulate`・`ingest`・`lint`
- **改善案**: 「架空/シミュレーションデータ由来の確信度は上限8」を確信度表に明記。lint が「sources が架空明示のACTに紐づく確信度が9-10」を検出。

## #4 frontmatter の配列参照（hypotheses / based-on）の接頭辞規約が曖昧 ／ 優先度: 中
- **摩擦**: `templates/activity.md` の `hypotheses: [H-NNN]`・`templates/decision.md` の `based-on: [ACT-NNN]` は接頭辞なし例。一方 `id` は接頭辞必須。整合が取れず、今回は `[SELF-H-001, ...]` と接頭辞つきで統一したが根拠が曖昧。
- **該当**: `templates/activity.md`・`templates/decision.md`／`lint`（ID整合）
- **改善案**: テンプレの配列例を接頭辞つき（`[SELF-H-001]`）に統一。lint が frontmatter 配列の接頭辞欠落を検出。

## #5 デスクリサーチを初期確信度の根拠にするワークフローが未定義 ／ 優先度: 中
- **摩擦**: 今回は「デスクリサーチ(実データ)を sources に置き→ desk-research ACT を作り→ それを根拠に仮説を確信度3-4で起票」という順序で進めた。これが `formulate`（起票）と `ingest`（ACT作成）のどちらの管轄か、初期確信度行の活動列に `[[ACT]]` を書く流儀があるか、明文化がない。
- **該当**: `formulate`・`ingest`
- **改善案**: 「デスクリサーチ先行→初期確信度3-4」のパターンを formulate/ingest の手順に位置づける。初期行の活動列を空(—)でなく desk-research ACT にできると根拠が明確になる。

## #6 プロジェクト丸ごと作り直し時の ID 採番方針が未定義 ／ 優先度: 中
- **摩擦**: 「ID再利用禁止・欠番として残す」は**生きているプロジェクト内**の規約。今回のように projects/self を丸ごと削除して作り直す場合、001から振り直してよいか（旧 self は 014 まで＋欠番002/003）が不明。今回は 001-008 で振り直した。
- **該当**: `CLAUDE.md`（ID採番規約）／`new-project`
- **改善案**: 「同名プロジェクトの作り直し（rebuild）時は ID を 001 から振り直してよい／旧IDは git 履歴に残る」を明記。または「作り直しは別 slug を推奨」とする。

## #7 new-project の cp が隠しディレクトリを巻き込む余地 ／ 優先度: 低
- **摩擦**: `cp -r templates/project/. projects/self/` 実行後、`projects/self/.obsidian/`（Obsidian が生成）が現れた。今回は `.gitignore` 済みで無害だったが、`/.` の挙動と環境依存ファイルの扱いが `new-project` SKILL に明示されていない。
- **該当**: `new-project`
- **改善案**: cp コマンドの正確な形と、コピー後に想定外ファイルが無いか確認する手順を SKILL に追記。

## #8 スキル改善（メタ観察）の記録先がスキーマに無い ／ 優先度: 低
- **摩擦**: 本ファイルのようなドッグフーディング所見の置き場所が定義されておらず、`projects/self/skill-improvements.md` にアドホックに作った。
- **該当**: `CLAUDE.md`（3層アーキテクチャ）／`lint`
- **改善案**: メタ観察の標準的な置き場所（例 `projects/<slug>/skill-improvements.md`、または schema層に `IMPROVEMENTS.md`）を定義。`lint` が実行時に気づいた改善候補をここへ追記する運用も検討。
