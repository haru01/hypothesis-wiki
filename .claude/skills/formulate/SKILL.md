# /formulate — アイデアを反証可能な仮説に精錬する

曖昧なアイデアを、確信度とステータスを持つ検証可能な仮説レコードに変える。**一度に1問ずつ**深掘りする。

> **パス・ID規約**: 以下の `sources/` `wiki/` は現在のプロジェクト `projects/<slug>/` 配下を指す（現在のプロジェクトは `projects/current.md` の `current-project`）。ID とファイル名は**プロジェクト接頭辞つき**（例 `SELF-H-001`）。採番は種別×プロジェクトごとの既存最大+1。

## 手順

1. **現在ステージを読む** — `wiki/stage.md` で現在ステージを確認し、対応する `playbooks/<stage>.md` の「中心の問い（問いかけバンク）」と「形成すべき仮説タイプ」を読む。

2. **問いかけバンクから1問ずつ聞く** — `wiki/index.md` を見て、まだ答えられていない／仮説化されていない問いを選ぶ。`AskUserQuestion` で1問ずつ問う。回答が曖昧なら、反証可能になるまでさらに1問深掘りする。ソリューションの話に流れがちなCPF/FPFでは「顧客と課題」に引き戻す。
   - **順序**: CPF/FPFでは**まず状況・行動仮説を1本確定させてから課題仮説に進む**。課題（痛み）へ早く流れると行動の事実を押さえ損ねる。
   - 課題（CPF Q4以降）に入る前に、**行動を尋ねる問い（CPF Q3「その状況で顧客は現在どんな行動をとっているか」）を必ず1問挟む**。一般論が出たら「直近の具体例は？」で動作レベルまで降ろす。

3. **反証可能な仮説文に整形する** — 「〈顧客〉は〈状況〉において〈課題／行動／価値〉である」の形式で、何が起きれば反証かが明確な一文にする。検証不能な言い切り（「重要だと思う」等）は書き換える。

4. **タイプ・初期確信度・ステータスを決める** — タイプは playbook の「形成すべき仮説タイプ」から判定。新規で証拠がなければ確信度は 1-2（勘）、状況証拠があれば 3-4。ステータスは通常 `未検証`。`importance` は原則 `auto`（ステージから自動決定）。**核心となる仮説なら frontmatter `core: true` を立てる**（list で ★ 表示）。

5. **レコードを発番・作成する** — `wiki/hypotheses/` の既存最大番号+1で `H-NNN` を採番（再利用禁止）。`templates/hypothesis.md` に従って作成。frontmatter の `id` は**ファイル名と同じ接頭辞つき**にする（例 ファイル `SELF-H-001.md` なら `id: SELF-H-001`）。**frontmatter `short-title` に list の mermaid 用の短ラベル（8字程度）を書く**（省略時はタイトルを機械切り詰め）。派生元があれば本文の「系譜」に `[[H-NNN]]` を書く。**この仮説が因果的に別の仮説を導くなら frontmatter `leads-to` に接頭辞つきで列挙し、本文「系譜」にも wikilink を併記する**（list の mermaid 矢印になる。frontmatter だけではObsidianグラフに辺が出ない）。**ソリューション仮説なら frontmatter `addresses` に対応する課題仮説を列挙する**（vp のフィット表になる）。確信度履歴テーブルに初期行を入れる。ディレクトリに `.gitkeep` が残っていれば削除してよい。

6. **index と log を更新する** — `wiki/index.md` の該当タイプ表に1行追加。`wiki/log.md` に追記:
   `## [YYYY-MM-DD] hypothesis | H-NNN <要約> → 新規作成 確信度N/未検証`

7. **セッション終了時の自己点検** — 「状況・行動仮説が最低1件あるか」を確認する。課題仮説ばかりで行動仮説が0件なら、行動（CPF Q3）の深掘りに戻して1本作る。

## 守ること

- 一度に複数の仮説を量産しない。1本ずつ丁寧に精錬する。
- 確信度は「勘」レベルから始めてよい。低い確信度を恐れず正直に付ける。
- 仮説文は必ず**反証可能**にする。反証条件を言えない文は仮説ではない。

## 仮説の取り下げ

ユーザーが「この仮説は取り下げたい／忘れて」と言ったときの手順（スキーマは変えず運用で扱う）。

- **反証との区別**: 起票直後・検証活動（ACT）に未紐づけの取り下げは**反証ではない**（証拠で崩れたわけではない、単なる撤回）。検証活動に紐づいて否定された場合は取り下げでなく `/ingest` の**反証**（status `反証`）で扱う。
- **手順**:
  1. 対象レコード `wiki/hypotheses/H-NNN.md` を削除する。
  2. `wiki/log.md` に追記: `## [YYYY-MM-DD] hypothesis | H-NNN 取り下げ（ユーザー判断） → レコード削除。反証ではない。ID H-NNN は欠番として再利用しない`
  3. **IDは欠番として残し、再利用しない**（不変ルール5）。詰め直し（リナンバリング）はしない。
  4. 他レコード本文の被リンク `[[H-NNN]]` を洗い出し（`grep`）、残る仮説へ繋ぎ直すか記述を修正する（リンク切れを残さない）。
  5. `index.md` の該当行を削除。生成ビューは `/view` で再生成する。
