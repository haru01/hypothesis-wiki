# /plan — 次に検証すべき仮説の抽出とテストカード立案

「重要度が高く・確信度が低く・未検証/検証中」の仮説（アサンプションマッピングの「重要×証拠なし」象限）を選び、検証計画を立てる。

> **パス・ID規約**: 以下の `sources/` `wiki/` は現在のプロジェクト `projects/<slug>/` 配下を指す（現在のプロジェクトは `projects/current.md` の `current-project`）。ID とファイル名は**プロジェクト接頭辞つき**（例 `SELF-ACT-001`）。採番は種別×プロジェクトごとの既存最大+1。

## 手順

1. **状況を読む** — `wiki/stage.md`（現在ステージと移行基準の上書き）、`wiki/index.md`（全仮説）、対応する `playbooks/<stage>.md`（重点仮説タイプ・適した検証手法）を読む。

2. **重要度を解決する** — 各仮説の `importance` が `auto` なら CLAUDE.md の「ステージ→重点仮説タイプ」で解決（重点タイプ=8、それ以外=4）。手動指定（1-10）があればそれを優先。

3. **重要度×証拠の2軸マップを提示する** — 仮説を「重要度（高/低）× 確信度（低/高）」の4象限に配置して見せる。**優先すべきは「重要度高 × 確信度低 × 未検証/検証中」**。

4. **検証手法を提案する** — 優先仮説に対し、playbook の「適した検証手法」から具体的な方法を提案する。ユーザーにどの仮説をどの方法で検証するか選んでもらう（`AskUserQuestion`）。

5. **テストカード付き ACT 草稿を作る** — 選ばれた仮説ごとに `wiki/activities/` の既存最大+1で `ACT-NNN` を採番し、`templates/activity.md` の**テストカード部分のみ**（目的・方法・指標・成功基準）を記入して作成。frontmatter の `id` は**ファイル名と同じ接頭辞つき**にする（例 `id: SELF-ACT-001`）。学習カードは空のまま（検証後に `/ingest` で埋める）。本文に対象仮説 `[[H-NNN]]` を書く。ディレクトリに `.gitkeep` が残っていれば削除してよい。
   - **interview/demo 系はスクリプト草稿も併せて作る** — `templates/interview-script.md` を基に、対象仮説を差し込んだ現場用スクリプト草稿を `wiki/activities/<PREFIX>-ACT-NNN-script.md` として作成し、ACT本文から参照する。CPFの鉄則（過去の事実を聞く／ソリューションを見せない／誘導しない）と、対象者ごとの〈自認〉〈実コスト〉○×記録シートを含める（実例: `projects/self/wiki/activities/SELF-ACT-001-script.md`）。

6. **log を更新する** — `wiki/log.md` に追記:
   `## [YYYY-MM-DD] <type> | ACT-NNN テストカード作成（<対象H-NNN>） → 検証計画`
   （type は活動タイプ interview/demo/... のいずれか）

## 守ること

- **成功基準は検証開始前に確定する**。後から書き換えない（後知恵バイアス防止）。
- 一度に検証しすぎない。最重要の1-2本に絞るよう促す。
- この段階では確信度・ステータスを変更しない（それは `/ingest` の仕事）。
