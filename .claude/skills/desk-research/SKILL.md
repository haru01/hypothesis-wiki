# /desk-research — 対象ドメイン・競合の調査と仮説の起票

対象ドメインを実Web検索で調べ、想定ユーザの状況・行動/課題仮説を生成し、競合のアプローチを調査・比較する。
発見は**出典付き生データとして `sources/` に残し**、desk-research 活動レコード（ACT）を作り、行動/課題仮説を
**確信度3-4（二次情報の上限）で起票（提案 → ユーザー承認 → 作成）**する。一次インタビュー前に相場観を掴む入口スキル。

> **共通規約**（プロジェクト解決・ID/接頭辞・リンク記法と `../` 深さ・.gitkeep・承認規律）は [CLAUDE.md「スキル共通規約」](../../../CLAUDE.md) が正典。以下は本スキル固有の手順。

## 手順

1. **現在プロジェクト・ステージを解決**（→ 共通規約1）。対応する `playbooks/<stage>.md` の「中心の問い（問いかけバンク）」と「形成すべき仮説タイプ」を読む（デスクリサーチが主に効くのは CPF/FPF）。

2. **調査スコープを決める** — `AskUserQuestion` で調査の対象を確認する:
   - 何を調べるか（①想定ユーザの状況・行動/課題 ②競合のアプローチ比較 ③両方）。
   - 対象ドメイン／顧客セグメント／主要キーワード（絞れていなければここで具体化する）。
   - `wiki/index.md` と `wiki/hypotheses/` を先に読み、**既にカバー済みの仮説と重複しない**空白を狙う。

3. **実Web検索で一次情報を集める** — `WebSearch`・`WebFetch` で複数角度から調べ、**各主張に出典URLを必ず紐づける**。
   - **想定ユーザ側**: 誰が・どんな状況で・どんな行動をとり・どこでつまずき・代替手段は何か・そのコストは（`playbooks/cpf.md` Q1–Q7 に沿う）。**独立した複数の信頼できる出典で三角測量**する。単一出典・伝聞のみは弱いと控える。
   - **競合側**: 競合／代替手段を洗い出し、各社の一次ページ・ドキュメント・READMEを `WebFetch` し、「重なる核」「決定的な違い」「出典URL」「時点情報」を集める（`docs/competitive-analysis.md` の競合マップ表が模範）。

4. **生データを sources/ に記録する（不変）** — 発見を構造化して `sources/` に置く。**一度置いた `sources/` は以後編集しない。**
   - 想定ユーザ: `sources/YYYY-MM-DD-desk-research-<topic>.md`。冒頭に「**データ種別: デスクリサーチ（二次情報・状況証拠）。一次インタビューは未実施**」を明記。状況・行動 S1..／課題 P1..（各に出典URLと「なぜ起きるか」）／課題のクラスタ／引用文献。（実例: `projects/self/sources/2026-07-16-desk-research-corporate-hypothesis-testing.md`）
   - 競合: `sources/YYYY-MM-DD-desk-research-competitors-<topic>.md`。競合マップ（競合｜系統｜重なる核｜決定的な違い｜出典）／時点情報／誠実な但し書き（一次ベンダーページはマーケ色を帯びる・不在系の主張は機能ページから断定できない）。

5. **desk-research 活動レコード（ACT）を作る** — `wiki/activities/` の既存最大+1で `<PREFIX>-ACT-NNN` を採番。`templates/activity.md` に従い `type: desk-research` で作成する。frontmatter の `id` は**ファイル名と同じ接頭辞つき**（例 `id: SELF-ACT-001`）。ディレクトリに `.gitkeep` が残っていれば削除してよい。
   - **テストカード**（検証前に記入・後から書き換えない）: 目的／方法（Web検索＋一次ページの突き合わせ）／指標（同一の状況・課題を指摘する独立出典の数と一致度／競合なら比較軸のカバー）／**成功基準**（複数の独立した信頼できる出典が同一の状況・課題を指摘していれば確信度3-4で起票。単一出典・伝聞のみは確信度3止まり）。**frontmatter `riskiest-assumption`** に「最もリスクの高い前提」（例: この課題は文献の一般論でなく特定の作り手が実コストを払っている）を一文で記入する（board の背骨）。`projects/self/wiki/activities/SELF-ACT-001.md` のテストカードが模範。
   - **学習カード**（検証後に記入）: **学びの要点**（一行の見出し的学び。board へ射影）・事実（想定ユーザ=S/P を出典付きで／競合=競合マップ表）・解釈（課題を3クラスタ等に整理／差別化の論点＝勝ち筋・代替手段。事実と明確に分ける）・驚き／想定外・確信度の更新テーブル・次のアクション（一次インタビューで「特定顧客が実際に実コストを払っているか」を確認する）。frontmatter `outcome` は `起票`（二次情報での新規起票）を記入する。
   - 本文冒頭に `対象仮説: [[<PREFIX>-H-NNN]] …`（wikilink）を書き、frontmatter `hypotheses` は接頭辞つき配列（例 `[SELF-H-001, SELF-H-002]`）。生データへは相対mdリンクで参照する。

6. **行動/課題仮説を起票する（提案 → 承認 → 作成）** — リサーチから浮かぶ仮説を**提案として提示し、ユーザーの承認を得てから**作成する。
   - **順序**: CPF/FPF では**まず状況・行動仮説を確定させてから課題仮説**に進む。ソリューションの話に流れない。
   - **文面**: 「〈顧客〉は〈状況〉において〈行動/課題〉である」の反証可能な一文。**generic な言い切りを避け、特定顧客・特定状況・特定行動まで具体化する**（浅い仮説は響かない）。何が起きれば反証かを明確にする。
   - タイプは playbook の「形成すべき仮説タイプ」から判定。`importance` は原則 `auto`。
   - 承認後、`wiki/hypotheses/` の既存最大+1で `<PREFIX>-H-NNN` を採番（再利用禁止・欠番は残す）。`templates/hypothesis.md` に従い作成し、`id` は**ファイル名と同じ接頭辞つき**。
   - **確信度は 3-4**（二次情報の上限。単一出典・伝聞なら 3）。確信度履歴テーブルの初期行の**活動列に、この desk-research ACT の `[[<PREFIX>-ACT-NNN]]` を書く**（初期確信度の根拠を明示する）。根拠列の先頭に証拠種別タグ〈二次〉を付ける（例 `〈二次〉独立3出典が…`）。
   - 競合調査由来のときは、判明した代替手段を課題仮説本文の「代替手段」に、勝ち筋を FPF の**自分たち仮説**に反映し、いずれもこの ACT に紐づける。

7. **相互リンクを張る** — ACT 本文に対象仮説の `[[<PREFIX>-H-NNN]]`、各仮説の系譜/確信度履歴に `[[<PREFIX>-ACT-NNN]]` を書く（frontmatter 配列だけにしない）。schema層（`playbooks/` など vault 内の接頭辞つきノートでないファイル）への参照は**相対mdリンク**で書く（wikilink は解決しない）。

8. **index と log を更新する** — `wiki/index.md` の該当タイプ表に新仮説の行を追加。`wiki/log.md` に**追記のみ**で1行足す:
   `## [YYYY-MM-DD] desk-research | ACT-NNN <要約> → H-NNN..H-NNN 新規起票 確信度3-4/未検証`

## 守ること

- デスクリサーチ由来の確信度は**上限3-4**（二次情報・状況証拠）。9-10は実観測のみ。実Web検索の**出典URLを必ず残す**。
- 出典のない主張を書かない。単一出典・伝聞は確信度3止まりで「弱い」と明記する。**不在系の主張（「競合にXが無い」）は機能ページから断定できない**と但し書きする（`docs/competitive-analysis.md` の誠実な但し書きに倣う）。競合の一次ソースは**時点情報つき**で残す（機能は急速に陳腐化する）。
- 仮説は鋭く具体的に。generic な言い切りを避け、特定顧客・状況・行動まで降ろす。
- CPF/FPF ではソリューションの話に流れず顧客と課題に集中する。状況・行動仮説を最低1本は確定させる。
- 確信度の付与（起票の初期確信度も含む）は**必ずこの ACT に紐づける**。根拠レコードなしに書き換えず、承認なしに量産しない。
- `sources/` は一度置いたら編集しない（不変ルール3）。テストカードの成功基準は検証開始後に書き換えない（後知恵バイアス防止・不変ルール6）。
