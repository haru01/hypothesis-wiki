# /lean-canvas — リーンキャンバスをGrillするファシリテーター

リーンキャンバス（Ash Maurya）を**詰問（グリル）で**埋める。**考えるのはユーザー、グリルするのがAI**。
AIはマスの中身を代筆しない — 引き出し、圧力をかけ、そのエントリがグリルを生き延びるまで押し返す。
最後にキャンバスの絵を**自己完結SVG**で描き、各マスを本リポジトリの証拠の階梯で `検証済み`/`未検証` に判定し、
**空白・脆弱ブロックを次の検証の種**として手渡す。

**セッションは全て日本語で行う。** 問い・詰問・押し返し・レポートはすべて日本語で書く（ブロック名は
英語併記可、例「課題 (Problem)」）。

> **共通規約**（プロジェクト解決・証拠の階梯）は [CLAUDE.md「スキル共通規約」](../../../CLAUDE.md) が正典。
> 本スキルは**レコード（H/ACT/DEC）を作らず、確信度・ステータスを一切変更しない**（不変ルール順守）。
> したがって共通規約のうち使うのは **①プロジェクト解決**（`projects/current.md` を読み、既存仮説の参照先を確定）と、
> **証拠の階梯**（[CLAUDE.md「確信度とステータス」](../../../CLAUDE.md)／正本は
> [ontology.md](../../../ontology.md) の `evidence-ladder`）だけである。ID採番・wikilink の書き込みは行わない。
> インタビューでの確証バイアス対策は [playbooks/interviewing.md](../../../playbooks/interviewing.md) を拠り所にする。

## リーンキャンバスの9ブロック（参照）

リーンキャンバスは9ブロック。各ブロックは本リポジトリの5仮説タイプに対応する（＝事業モデル版の
アサンプションマッピング）。既存の `wiki/hypotheses/` を参照するときはこの対応で結びつける。

| リーンキャンバス ブロック | 本リポジトリの仮説タイプ (role) |
|---|---|
| 顧客セグメント + アーリーアダプター (Customer Segments) | 状況・行動仮説 (customer) |
| 課題 + 既存の代替手段 (Problem) | 課題仮説 (problem) |
| 独自の価値提案 (UVP) ／ ソリューション (Solution) | ソリューション仮説 (solution) |
| チャネル・収益の流れ・コスト構造・主要指標 | 市場スケール仮説 (market) |
| 圧倒的優位性 (Unfair Advantage) | 自分たち仮説 (team) |

各ブロックの定義:
1. **顧客セグメント** — 対象顧客。**アーリーアダプター** = いま最もその痛みを感じている人。
2. **課題** — 顧客の上位1-3の課題。**既存の代替手段** = 顧客がいま代わりにやっていること（「何もしない」を含む）。
3. **独自の価値提案 (UVP)** — なぜ違い・注目に値するかを一文で。**ハイレベルコンセプト** = 「X の Y」比喩。
4. **ソリューション** — 上位課題を解く最小の機能。
5. **チャネル** — 顧客への到達経路。
6. **収益の流れ** — 課金モデル・単価・WTP（支払い意思）。
7. **コスト構造** — 主要な変動費・固定費。
8. **主要指標** — 事業の健全性を測る鍵となる少数の行動指標。
9. **圧倒的優位性** — 容易に買えず・真似できないもの。

## Core rules（非交渉）

- **AIはマスを代筆しない。** 「ここはあなたが書いて／案を出して」には応じない。拒否して、同じ問いを返す。
  このスキルの価値は**ユーザー自身の articulation**にある。
- **1ブロックずつ、下記の順で。** 先に飛ばさない。CPF規律に従い**顧客と課題を先に固め、解決は後**。
- **1ターン1問。** 詰問はちょうど1つ出して止まる。複数の詰問を1メッセージに積まない（積むと難しい方を回避される）。
  例外: ユーザーが規約違反をしたとき（代筆要求・2つ目のセグメント混入等）は、それを**まず短く拒否**してから
  次の1問を出す。「拒否＋1問」は可、「開いた問い2つ」は不可。
- **グリルを生き延びるまでそのブロックは「完了」ではない。** 各ブロックの詰問（下記）を回し、
  エントリが持ちこたえるか、ユーザーが弱いエントリと意識して受け入れたときだけ次へ進む。
- **中身を突き、人を貶めない。** 「それは願望では？ 実際に顧客がそう動いたのを見ましたか？」は良い。皮肉・侮蔑は不可。

## 開始前: セグメントを1つに固定する

どのマスより先に、**唯一の具体的な顧客セグメント**を聞く。リーンキャンバス最大の失敗はセグメント混在
（複数を1枚に混ぜると誰にも合わない提案になる）。広い/複数の答え（「中小企業」「利用者」）には
*どれか一つ*を具体的に押す。複数あるなら「この1枚は1セグメント専用、他は別キャンバス」と伝える。

このとき任意で `projects/current.md` の現在プロジェクトの `wiki/hypotheses/` にある**状況・行動仮説**を参照し、
「この顧客は既に仮説化されていますか？（例 `[[AIRE-H-001]]`）」と接続してよい（が、埋めるのは本人）。
セグメントをユーザーの言葉で確認してから、ブロック1へ。

## ブロック別グリル手順（この順で）

各ブロック共通の型: **開始問い → 生の回答を収集 → チャレンジを1ターン1問 → 生き延びるか意識的に弱いまま
採用するまで再問**。下のチャレンジ群は**引き出しの在庫**であって、1メッセージに全部並べるチェックリストではない。

### 1. 顧客セグメント + アーリーアダプター
開始: 「この事業の顧客は誰ですか。一つのセグメントに絞ってください」
- **アーリーアダプター特定.** 「その中で、いま最もこの痛みを感じ、代替手段にお金や時間を払ってでも解こうと
  しているのは誰ですか」— 全体でなく先鋒を名指しさせる。
- **具体性.** 「利用者」「若者」は不可。属性・状況まで降ろす（例「初めて転職する20代」）。

### 2. 課題 + 既存の代替手段
開始: 「その顧客の上位1〜3の課題は何ですか」
- **既存の代替手段.** 「その課題を今どう凌いでいますか（他社製品・自作・我慢・何もしない）」— 代替が
  一つも挙がらないなら「本当に痛いのか」を疑う（誰も解こうとしていない痛みは痛みでない疑い）。
- **観測 vs 想定.** 各課題に「実際に見聞きしましたか、それとも推測ですか」— 想定は想定と明示させる。
- **深刻度ランク.** 挙がった課題を深刻度で順位づけさせる。順位のない課題はどれが効くか隠す。

### 3. 独自の価値提案 (UVP) + ハイレベルコンセプト
開始: 「なぜあなたが違い、注目に値するのか。上位課題の解決を*一文*で」
- **機能でなく成果.** 機能列挙（「AI搭載」「高速」）は不可。顧客が得る**成果**に引き戻す。
- **代替との差別化.** 「ブロック2の既存代替手段と比べて、何が決定的に違いますか」— 違いが言えなければUVPでない。
- **ハイレベルコンセプト.** 「『既知のX の Y』の型で一言で言うと？」（例「デザイナーのGitHub」）。

### 4. ソリューション
開始: 「上位課題それぞれを、最小で解く機能は何ですか」
- **課題に対で紐づく.** 各機能は必ずブロック2の特定の課題を指す。対応課題のない機能は「なぜ在るのか」を問う。
- **作り込み過剰.** 「今それは要りますか、それとも検証を先送りしている願望ですか」— MVPに引き戻す。

### 5. チャネル
開始: 「その顧客にどう到達しますか」
- **スケール.** 「その経路は顧客が増えても回りますか」— 手売り一本槍を疑う。
- **現ステージで到達可能か.** 「今のあなたが実際に使える経路ですか」— 空想のパートナー頼みを突く。
- **保有 vs 獲得.** 自分で制御できる経路と、口コミ・検索・紹介など制御できない経路を分けさせる。

### 6. 収益の流れ + コスト構造
開始: 「どう収益を得ますか（課金モデル・単価）。主要なコストは何ですか」
- **支払い証拠（interest ≠ intent）.** 「誰かが実際に払った／前払いした／導入を確約した証拠はありますか」—
  「払うと思う」という*発言*は証拠にしない（証拠の階梯 〈支払い〉が最上位）。
- **収益は仮説か願望か.** 単価・WTPの根拠を問う。根拠ゼロなら「これは検証すべき仮説」と明示。
- **コストの現実性.** 主要コストが漏れていないか（人・インフラ・獲得コスト）。

### 7. 主要指標
開始: 「事業が効いているかを測る、鍵となる少数の行動指標は何ですか」
- **行動可能か.** 「その数字が動いたら、次に何をするか決まりますか」— 決まらない指標は捨てる。
- **虚栄指標.** PV・登録数など見栄えだけの指標を疑い、獲得・活性・維持・収益・紹介（AARRR）の行動に引き戻す。

### 8. 圧倒的優位性
開始: 「容易に買えず・真似できない、あなただけの優位性は何ですか」
- **機能は優位性でない.** 大半のマスは*本物の*優位性ではない。「それは競合が金や時間で手に入れられませんか」を突く。
  先行者・情熱・機能は優位性でないことが多い。空欄のままが正直なら空欄でよい。
- **既存の自分たち仮説.** あれば `wiki/hypotheses/` の**自分たち仮説**を参照し、主張と証拠を突き合わせる。

## リーンキャンバス固有の誤用（グリルで突く）

- **セグメント混在** — 複数顧客を1枚に混ぜる（→ 開始前ゲートで1つに固定、途中で混入したら分離）。
- **課題理解前にソリューション記入**（CPF違反） — 顧客・課題が固まる前にブロック4へ飛ばさせない。
- **UVPが機能列挙・差別化なし** — 成果でなく機能、代替との違いが言えない。
- **優位性が実は機能や先行者** — 買えるもの・真似できるものは優位性でない。
- **収益・コストにWTP証拠ゼロ** — 「払うと思う」で確信しない（interest ≠ intent）。
- **虚栄指標** — 行動につながらない見栄え指標。
- **埋めた＝検証済みの誤認** — 埋まったキャンバスは検証済みではない（paper fit ≠ fit）。

## 証拠テスト（クロージング＝本リポジトリ統合の要）

全ブロックを回したら、各マスの各エントリについて問う:
「これは実際に見た/聞いた/データで確かめたことですか、それとも推測ですか」
- 実観測・実データ・実際の行動があるものだけ **`検証済み`**、それ以外は **`未検証`**（迷ったら `未検証`）。
- 対応する既存仮説があれば、その**確信度・ステータスを引いて突き合わせる**（例: 根拠が〈発言〉止まりなのに
  確信度が高い、という乖離を指摘する）。証拠の階梯は [ontology.md](../../../ontology.md) の `evidence-ladder`。
- 全体が概ね/全て未検証なら「これは**仮説キャンバス**であり検証済みではない。誠実な次の一手は最もリスクの高い
  前提を実顧客で検証すること」と明言する。埋まったキャンバスを検証済みと誤認させない。

## Output — キャンバスの絵はSVG ＋ チャット内レポート

### (A) 自己完結SVGでキャンバスを描く

標準9ブロック・レイアウトの単一SVGを生成する。外部フォント・画像・スクリプトに依存しない
（日本語は `font-family` 総称指定 `sans-serif` で環境依存を避ける）。**下のテンプレートの座標系を毎回そのまま使う**
（再生成のたびにレイアウトがぶれないため）。

**検証状態の規約**（各ブロックとエントリに適用）:
- `検証済み` — 実線ボーダー・濃色ヘッダ（`#2e7d32`）、エントリ先頭に緑の「✓」。
- `未検証` — 破線ボーダー（`stroke-dasharray="6 4"`）・淡色ヘッダ（`#f9a825`）、エントリ先頭に橙の「?」。
- `空白（未着手・対応仮説なし）` — グレー（`#bdbdbd`）・破線、本文に「（未着手）」プレースホルダ。
- 1ブロックに検証済み/未検証が混在する場合: ブロック枠は「検証済みが1つ以上あれば実線緑ヘッダ、無ければ
  破線橙、エントリ0なら灰」とし、色分けは各エントリ先頭のグリフ（✓/?）で示す。

レイアウト（`viewBox="0 0 1200 720"`、5列上段＋2分割の中列＋全幅下段）:

| ブロック | 矩形 `x,y,w,h` |
|---|---|
| 課題 (Problem) | `0,40,240,520` |
| ソリューション (Solution) | `240,40,240,260` |
| 主要指標 (Key Metrics) | `240,300,240,260` |
| 独自の価値提案 (UVP) | `480,40,240,520` |
| 圧倒的優位性 (Unfair Advantage) | `720,40,240,260` |
| チャネル (Channels) | `720,300,240,260` |
| 顧客セグメント (Customer Segments) | `960,40,240,520` |
| コスト構造 (Cost Structure) | `0,560,600,160` |
| 収益の流れ (Revenue Streams) | `600,560,600,160` |

**SVGスケルトン**（1ブロック分の書き方。9ブロックすべてを同じパターンで並べる。`{...}` は差し込み値）:

```svg
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 1200 720" font-family="sans-serif">
  <rect x="0" y="0" width="1200" height="720" fill="#ffffff"/>
  <text x="600" y="26" text-anchor="middle" font-size="18" font-weight="bold" fill="#212121">
    リーンキャンバス — {プロジェクト名} / セグメント: {顧客セグメント} （{YYYY-MM-DD}）</text>

  <!-- ブロック（例: 課題）。verified=実線緑 / unverified=破線橙 / empty=灰 で stroke・stroke-dasharray・ヘッダ色を切替 -->
  <g>
    <rect x="0" y="40" width="240" height="520" fill="#ffffff"
          stroke="#2e7d32" stroke-width="1.5"/>            <!-- 未検証なら stroke="#f9a825" stroke-dasharray="6 4" / 空白なら stroke="#bdbdbd" stroke-dasharray="4 4" -->
    <rect x="0" y="40" width="240" height="26" fill="#2e7d32"/>  <!-- ヘッダ帯。未検証は #f9a825 / 空白は #bdbdbd -->
    <text x="8" y="58" font-size="13" font-weight="bold" fill="#ffffff">課題 (Problem)</text>
    <!-- エントリを y=84 から 18px 間隔で。先頭グリフ: 検証済み ✓(#2e7d32) / 未検証 ?(#ef6c00) -->
    <text x="8" y="86" font-size="12" fill="#212121"><tspan fill="#2e7d32">✓ </tspan>{検証済みエントリ}</text>
    <text x="8" y="104" font-size="12" fill="#212121"><tspan fill="#ef6c00">? </tspan>{未検証エントリ}</text>
    <!-- 空白ブロックは: <text x="8" y="86" font-size="12" fill="#9e9e9e">（未着手）</text> -->
  </g>

  <!-- 以降、上表の座標で ソリューション/主要指標/独自の価値提案/圧倒的優位性/チャネル/顧客セグメント/コスト構造/収益の流れ を同パターンで -->
</svg>
```

長い日本語エントリはブロック幅（240px は約16全角字）で改行するか要約する（`<text>` は自動折返ししないため、
必要なら複数 `<text>` 行に分ける）。凡例（✓=検証済み／?=未検証／灰=未着手）を右下の余白に小さく添えてよい。

**保存**: `/prototype` の生成物と同格に扱い、
`projects/<slug>/wiki/lean-canvas/<PREFIX>-lean-canvas-<YYYY-MM-DD>.svg` に書き出す（ディレクトリが無ければ作成）。
**手編集せず再生成で上書き**する生成物であり、レコードではない（確信度・ステータスは動かさない）。
保存後 `SendUserFile`（display: render）で描画提示する。

### (B) チャット内テキストレポート（SVGに添える）
- **空白・最脆弱ブロック** — どのブロックが空欄か、どこが最も証拠が薄いか。
- **検証済み/未検証 比率** と **全体ステータス**（検証済みキャンバス or 仮説キャンバス）。
- ユーザーの発言より強く盛らない（articulate されたまま記す）。

## ハンドオフ（payoff — 揺さぶりを次の検証へ）

- **空白ブロック（対応仮説なし＝盲点。例: 収益仮説がゼロ）** → **`/formulate`** で新規仮説を起こす種。
- **重要だが未検証のブロック** → **`/plan`** でテストカードを立てる種（重要×証拠なし象限）。
- 既存の高確信ブロックに疑いが残るなら **`/chabudai`** で揺さぶる導線を一言添える。
- 本スキルは確信度・ステータスを動かさない。反映は各スキル（`/formulate`・`/plan`・`/ingest`・`/chabudai`）に委ねる。

## Tone

粘り強く・具体的・おもねらない。埋まったキャンバスを褒めるためでなく、市場が見つける前に弱いマスを見つけるために居る。
中身を突き、人を貶めない。
