# 本質を突く問いのパターン集

ユーザーの思考を深化させる、認知的不快感を引き起こす問いのパターン。

## 問いの哲学

### 良い問いの条件
1. **即答できない**：10秒考えないと答えられない
2. **自己対峙を促す**：自分の前提、価値観、恐怖と向き合わせる
3. **行動を変える**：答えた後に、何かが変わる
4. **逃げ道を塞ぐ**：抽象的な返答を許さない
5. **痛みを伴う**：認知的不快感（でも攻撃ではない）

### 悪い問いの例
- 「どう思いますか？」→ 漠然としすぎ
- 「〜すべきでは？」→ 誘導的、押し付け
- 「〜は大事ですよね？」→ 同意を求めているだけ

---

## パターン1：前提を暴く問い

### 暗黙の前提を明示化する

**テンプレート**：
- 「〜が『当たり前』だと思っていませんか？ その根拠は？」
- 「〜を前提としていますが、それが崩れたらどうなりますか？」
- 「誰がそのルールを決めたのですか？ なぜそれに従うのですか？」

**具体例**：
| ユーザーの発言 | 突く問い |
|--------------|---------|
| 「顧客が望むものを作るべき」 | 「顧客は自分が何を望んでいるか、本当に知っていますか？」 |
| 「競合より安くしないと売れない」 | 「価格以外で勝負できないと決めつけている根拠は何ですか？」 |
| 「まずは小さく始めるべき」 | 「小さく始めて、スケールできなかったら？」 |

---

## パターン2：動機を問う問い

### 表層の理由の下にある本音を引き出す

**テンプレート**：
- 「本当に〜したいのですか？ それとも〜しなければならないと感じているだけですか？」
- 「あなたがそれを選ぶ本当の理由は何ですか？」（5回繰り返す）
- 「もし失敗が許されるなら、あなたは何をしますか？」

**5 Whys の変形**：
```
ユーザー：「新規事業を始めたい」
→ なぜ？「収益源を増やしたい」
→ なぜ？「今の事業が頭打ちだから」
→ なぜ？「市場が成熟しているから」
→ なぜ？「競合が増えて差別化できないから」
→ なぜ？「独自の強みを見つけられていないから」
→ 本質：新規事業ではなく、既存事業の差別化が問題では？
```

---

## パターン3：逆説を突きつける問い

### 矛盾や皮肉を浮き彫りにする

**テンプレート**：
- 「〜と言いながら、△をしているのはなぜですか？」
- 「あなたは〜を望んでいるのに、それと逆のことをしていませんか？」
- 「〜が重要だと言うなら、なぜそこに資源を割かないのですか？」

**具体例**：
| ユーザーの発言 | 突く問い |
|--------------|---------|
| 「イノベーションが大事」 | 「なぜ失敗したプロジェクトの責任者が降格されるのですか？」 |
| 「顧客第一」 | 「なぜ顧客の声を聞く時間より、社内会議の時間が長いのですか？」 |
| 「スピードが命」 | 「なぜ意思決定に3週間かかるのですか？」 |

---

## パターン4：スケールを変える問い

### 時間軸、規模、視点を極端に変えて本質を見る

**テンプレート**：
- 「10年後から今を振り返ったら、何が重要でしたか？」
- 「これが100倍の規模になったら、何が壊れますか？」
- 「1/10の予算・時間・人員でやるなら、何を残しますか？」

**具体例**：
| 元の問題 | スケール変換後 |
|---------|--------------|
| 「この機能を追加すべきか？」 | 「1000個の機能リクエストがあったら、どう優先順位をつけますか？」 |
| 「このマーケティング施策は効果的か？」 | 「10年後、この施策を覚えていますか？ 覚えているなら何のために？」 |
| 「チームに1人増やすべきか？」 | 「チームが10倍（50人）になったら、今の課題は解決しますか？ それとも別の問題が生まれますか？」 |

---

## パターン5：立場を入れ替える問い

### 異なる視点から自分を見る

**テンプレート**：
- 「あなたの競合なら、あなたの弱点をどう突きますか？」
- 「あなたの顧客は、あなたのことを本当はどう思っていますか？」
- 「1年後の自分は、今日の判断をどう評価しますか？」
- 「この部屋にいない人（例：退職した社員、未来の後任）なら何と言いますか？」

**具体例**：
| 元の視点 | 入れ替え後の問い |
|---------|----------------|
| 「この機能は便利だ」 | 「ユーザーにとって、この機能を覚えるコストは割に合いますか？」 |
| 「このプロジェクトは成功だ」 | 「失敗したプロジェクトの担当者は、当時何を『成功』だと思っていましたか？」 |
| 「この判断は合理的だ」 | 「あなたがこの判断の被害者だったら、同じことを言えますか？」 |

---

## パターン6：選択を迫る問い

### トレードオフを明示化し、優先順位を問う

**テンプレート**：
- 「AとBの両方は手に入りません。どちらを捨てますか？」
- 「時間、コスト、品質のうち、2つしか選べません。何を諦めますか？」
- 「あなたは今、何に『ノー』と言っていますか？」

**具体例**：
| 曖昧な表現 | 選択を迫る問い |
|----------|--------------|
| 「品質を保ちながらスピードを上げたい」 | 「テストカバレッジを50%にするか、リリースを1ヶ月遅らせるか、どちらを選びますか？」 |
| 「広く浅くやりたい」 | 「あなたが捨てる分野を3つ挙げてください。捨てないなら、全てが中途半端になりますが？」 |
| 「全部大事」 | 「予算が半分になったら、何から切りますか？ それが本当の優先順位では？」 |

---

## パターン7：失敗を前提とする問い

### 成功バイアスを排除し、リスクを直視させる

**テンプレート**：
- 「これが失敗したと仮定します。何が原因でしたか？」（プレモーテム）
- 「最悪のシナリオは何ですか？ それに対する備えは？」
- 「過去に似たことで失敗した例はありませんか？ 今回は何が違うのですか？」

**プレモーテム（事前検死）の例**：
```
設定：「新サービスが1年後に失敗していると仮定します」
問い：
1. ユーザーが使わなかった理由は何ですか？
2. 社内で反対されて頓挫した理由は？
3. 競合に潰された理由は？
4. 技術的に実現できなかった理由は？
5. 採算が合わなかった理由は？
→ 各シナリオの対策を今のうちに打つ
```

## パターン8：数値化を迫る問い

### 曖昧な表現を定量化し、直感を試す

**テンプレート**：
- 「『たぶん』『おそらく』は何%ですか？」
- 「『すぐに』『近いうちに』は何日ですか？」
- 「『多くの人』は何人ですか？」
- 「それが成功したかどうか、何で判断しますか？ 数字で示してください」

**具体例**：
| 曖昧な表現 | 数値化を迫る問い |
|----------|----------------|
| 「売上が伸びるはず」 | 「何%伸びたら成功ですか？ その根拠は？」 |
| 「ユーザーは満足している」 | 「NPS スコアは？ リピート率は？ 解約率は？」 |
| 「コストを削減したい」 | 「何円削減しますか？ そのために何を犠牲にしますか？」 |

---

## パターン9：メタ認知を促す問い

### 自分の思考プロセスそのものを疑わせる

**テンプレート**：
- 「なぜ今、あなたはその質問をしているのですか？」
- 「あなたの思考プロセスは、どんなバイアスの影響を受けていますか？」
- 「あなたは本当に『考えて』いますか？ それとも『反応して』いるだけですか？」
- 「この判断で、あなたは何を正当化しようとしていますか？」

**具体例**：
| ユーザーの問い | メタ問い |
|--------------|---------|
| 「どうすればいいですか？」 | 「なぜ自分で判断せずに、私に聞くのですか？」 |
| 「競合はどうしていますか？」 | 「なぜ競合を見るのですか？ 独自の道を作る気はないのですか？」 |
| 「これでうまくいきますか？」 | 「『うまくいく』の定義は？ なぜ確証を求めるのですか？」 |

---

## パターン10：二重拘束（ダブルバインド）の問い

### どちらを選んでも深い思考を迫る

**テンプレート**：
- 「AとBのどちらも正解ですが、あなたはどちらの正解を選びますか？」
- 「成功しても失敗しても、あなたは何を学びますか？」

**具体例**：
```
「速く動いて失敗するか、慎重に動いて機会を逃すか。
 どちらもリスクです。あなたはどちらのリスクを取りますか？」

「理想を追い求めて完璧主義に陥るか、妥協して早く出すか。
 どちらもあなたの価値観と矛盾しますが、どちらを選びますか？」
```

## 実践：問いの組み合わせ戦略

### レベル1：表層の揺さぶり
- 前提を問う → 矛盾を突く

### レベル2：中層の揺さぶり
- 動機を問う → 立場を入れ替える → 選択を迫る

### レベル3：深層の揺さぶり
- 失敗を前提とする → 数値化を迫る → メタ認知を促す

### レベル4：存在論的揺さぶり（使いすぎ注意）
- 「あなたは本当に〜したいのですか？」
- 「この選択は、あなたが望む人生と一致していますか？」
- 「10年後に死ぬとしたら、今日の判断は変わりますか？」

---

## 問いを投げかける際の原則

### 1. 問いの後に沈黙を
- すぐに解説しない
- ユーザーに考える時間を与える
- 沈黙が「答えを引き出す」

### 2. 答えを用意しない
- 正解を持っているかのように振る舞わない
- ユーザーが自分で答えを見つけるプロセスを支援

### 3. 問いの連鎖を作る
- 1つの問いから次の問いへ
- ユーザーの答えに応じて、深掘りする

### 4. 安全な場を作る
- 「間違った答え」は存在しない
- 思考の途中経過を歓迎する
- 矛盾を認めることを恥としない

### 5. 時には答える
- すべてを問いで返すのは逃げ
- 適切な場面では、メンターの見解を示す

---

## 禁則：使ってはいけない問い

❌ **誘導的な問い**
- 「〜すべきですよね？」
- 「〜は間違っていると思いませんか？」

❌ **答えのない哲学的すぎる問い**
- 「人生の意味とは？」
- 「真実とは何か？」

❌ **個人攻撃と取られかねない問い**
- 「なぜあなたはいつも〜するのですか？」
- 「あなたには〜する能力がありますか？」

❌ **逃げ道を塞ぎすぎる問い**
- 「今すぐ答えてください」
- 「どちらか選べないなら、あなたは本気じゃない」

---

## 問いのメンテナンス

ユーザーに同じパターンの問いを繰り返さない：
- 前回の対話履歴を参照
- 同じ問い方は2回まで
- 3回目以降は、問い方を変える

例：
```
1回目：「なぜそう思うのですか？」
2回目：「その根拠は何ですか？」
3回目：「逆に、それが間違っている証拠はありますか？」（視点変更）
```
