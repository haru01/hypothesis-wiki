# /ingest — 生データの取り込みと学習カード作成・確信度更新

検証活動の生データを取り込み、事実と解釈を分離して学習カードに記録し、確信度・ステータスの更新を**提案 → ユーザー承認 → 反映**する。

> **パス・ID規約**: 以下の `sources/` `wiki/` は現在のプロジェクト `projects/<slug>/` 配下を指す（現在のプロジェクトは `projects/current.md` の `current-project`）。ID とファイル名は**プロジェクト接頭辞つき**（例 `SELF-H-001`・`SELF-ACT-001`）。採番は種別×プロジェクトごとの既存最大+1。

## 手順

1. **生データを sources/ に置く** — まだ置かれていなければ `sources/` に保存する（例 `sources/2026-07-16-interview-顧客A.md`）。**sources/ は読むだけ・以後編集しない。**

2. **対応する活動レコードを特定/作成する** — `/plan` で作った事前 ACT があればそれを使う。なければ `wiki/activities/` の既存最大+1で `ACT-NNN` を採番し `templates/activity.md` で新規作成（frontmatter の `id` は**ファイル名と同じ接頭辞つき**にする。例 `id: SELF-ACT-001`。ディレクトリに `.gitkeep` が残っていれば削除してよい）。この場合テストカードは事後補完でよいが、成功基準は当時の意図として正直に書く。

3. **学習カードを記入する** — 生データから:
   - **学びの要点**: まず一行（1-2文）で「この検証で最も重要な学び」を見出し的に鋭く書く（board の結果行に射影される）。事実の羅列でなく「何が言えたか」を書く。
   - **事実（observed）**: 観測した事実・発言を原文引用中心で。解釈を混ぜない。
   - **解釈（inference）**: 事実から何が言えるか。事実と明確に分ける。
   - **驚き・想定外**: 予想と違ったこと・新発見。
   - **課題ランキング検証では〈自認〉〈実コスト〉○×表を作る** — 対象者ごとに「自分の言葉で自認したか」「実コスト（時間/金/手戻り）を払っている証拠があるか」を○×で記録し、成功基準（切実な課題トップN）を機械的に判定する（実例: `projects/self/wiki/activities/SELF-ACT-001-script.md` の記録シート）。
   - **デスクリサーチで裏付ける** — 発見を `WebSearch` 等で三角測量し、傍証となる出典を仮説本文に残す（必要なら `desk-research` 活動としても別途記録）。
   - **取り込み中に仮説が創発したら、その場で起票する** — 事実・驚きから当初なかった行動/課題仮説が浮上したら、`/formulate` 相当で新規 `H-NNN` を起票し、この ACT に紐づける（実例 H-009/H-010）。ただし**事後発見は事前登録した検証ではない**ため、確信度を上げすぎない（専用の再検証を次アクションに積む）。

4. **確信度・ステータスの更新を提案する** — まずテストカードの成功基準を **〈支持／反証／判断保留〉の3値** で判定し、確信度・ステータスの更新提案はその判定に従わせる。**この判定を ACT frontmatter の `outcome` に永続化する**（`支持`／`反証`／`判断保留`。board サマリの outcome 列に射影される）。対象仮説ごとに「現在値 → 提案値・提案ステータス・理由」を提示する。**必ずユーザーの承認を得てから反映する**（AIが根拠なく確信度を変えない）。CLAUDE.md の確信度目安表に沿って提案する。
   - **成功基準が未達（反証・判断保留）なら原則として確信度を上げない**（据え置き、または下げる）。弱いシグナルがあって上げる場合は、その理由を成功基準の判定と切り離して明記する（部分証拠・架空データで確信度を水増ししない）。
   - **承認要否の段階化（非対話/バッチ実行向け）**: 対話セッションでは従来どおり承認を得る。自律実行・一括処理では、①成功基準の判定が機械的に〈支持〉または〈反証〉に定まり、②提案する確信度が証拠の階梯（`CLAUDE.md`）の範囲に収まる場合に限り、**提案内容を明示したうえで自動反映してよい**。〈判断保留〉や、成功基準の解釈を要する／証拠の階梯を超える引き上げを含む場合は、必ず対話で承認を得る（勘・部分証拠での水増しを自動化しない）。

5. **承認後に反映する** — 承認された仮説レコード（`wiki/hypotheses/H-NNN.md`）の frontmatter `confidence`/`status` を更新し、確信度履歴テーブルに1行追記（活動列に `[[ACT-NNN]]`、根拠列の先頭に証拠種別タグ〈発言〉〈自認〉〈実コスト〉〈行動〉〈支払い〉〈二次〉〈架空〉）。ACT の学習カードの確信度更新テーブルも埋める。

6. **index と log を更新する** — `wiki/index.md` の該当行を更新。`wiki/log.md` に追記:
   `## [YYYY-MM-DD] <type> | ACT-NNN <要約> → H-NNN 確信度X→Y/ステータス`

## 守ること

- 事実と解釈を必ず分ける。「顧客が言ったこと」と「私たちがそう思うこと」は別物。
- 確信度の変更は**必ずこの活動レコードに紐づく**。承認なしに書き換えない。
- 反証（成功基準を満たさなかった）の場合も正直に記録する。都合の良い解釈をしない。
- **架空/デモ・シミュレーションデータは必ず明示する** — 実インタビューでない場合、`sources/` の冒頭に「架空・実証拠として扱わない」と明記し、確信度履歴・ACT・ビューにも「実データ未検証」を残す。架空データで確信度を上げたまま実証拠と誤認させない。
