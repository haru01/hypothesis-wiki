# /ingest — 生データの取り込みと学習カード作成・確信度更新

検証活動の生データを取り込み、事実と解釈を分離して学習カードに記録し、確信度・ステータスの更新を**提案 → ユーザー承認 → 反映**する。

## 手順

1. **生データを sources/ に置く** — まだ置かれていなければ `sources/` に保存する（例 `sources/2026-07-16-interview-顧客A.md`）。**sources/ は読むだけ・以後編集しない。**

2. **対応する活動レコードを特定/作成する** — `/plan` で作った事前 ACT があればそれを使う。なければ `wiki/activities/` の既存最大+1で `ACT-NNN` を採番し `templates/activity.md` で新規作成（この場合テストカードは事後補完でよいが、成功基準は当時の意図として正直に書く）。

3. **学習カードを記入する** — 生データから:
   - **事実（observed）**: 観測した事実・発言を原文引用中心で。解釈を混ぜない。
   - **解釈（inference）**: 事実から何が言えるか。事実と明確に分ける。
   - **驚き・想定外**: 予想と違ったこと・新発見。
   - **課題ランキング検証では〈自認〉〈実コスト〉○×表を作る** — 対象者ごとに「自分の言葉で自認したか」「実コスト（時間/金/手戻り）を払っている証拠があるか」を○×で記録し、成功基準（切実な課題トップN）を機械的に判定する（実例: `wiki/activities/ACT-001-script.md` の記録シート）。
   - **デスクリサーチで裏付ける** — 発見を `WebSearch` 等で三角測量し、傍証となる出典を仮説本文に残す（必要なら `desk-research` 活動としても別途記録）。
   - **取り込み中に仮説が創発したら、その場で起票する** — 事実・驚きから当初なかった行動/課題仮説が浮上したら、`/grill` 相当で新規 `H-NNN` を起票し、この ACT に紐づける（実例 H-009/H-010）。ただし**事後発見は事前登録した検証ではない**ため、確信度を上げすぎない（専用の再検証を次アクションに積む）。

4. **確信度・ステータスの更新を提案する** — テストカードの成功基準に照らし、対象仮説ごとに「現在値 → 提案値・提案ステータス・理由」を提示する。**必ずユーザーの承認を得てから反映する**（AIが根拠なく確信度を変えない）。CLAUDE.md の確信度目安表に沿って提案する。

5. **承認後に反映する** — 承認された仮説レコード（`wiki/hypotheses/H-NNN.md`）の frontmatter `confidence`/`status` を更新し、確信度履歴テーブルに1行追記（活動列に `[[ACT-NNN]]`）。ACT の学習カードの確信度更新テーブルも埋める。

6. **index と log を更新する** — `wiki/index.md` の該当行を更新。`wiki/log.md` に追記:
   `## [YYYY-MM-DD] <type> | ACT-NNN <要約> → H-NNN 確信度X→Y/ステータス`

## 守ること

- 事実と解釈を必ず分ける。「顧客が言ったこと」と「私たちがそう思うこと」は別物。
- 確信度の変更は**必ずこの活動レコードに紐づく**。承認なしに書き換えない。
- 反証（成功基準を満たさなかった）の場合も正直に記録する。都合の良い解釈をしない。
- **架空/デモ・シミュレーションデータは必ず明示する** — 実インタビューでない場合、`sources/` の冒頭に「架空・実証拠として扱わない」と明記し、確信度履歴・ACT・ビューにも「実データ未検証」を残す。架空データで確信度を上げたまま実証拠と誤認させない。
