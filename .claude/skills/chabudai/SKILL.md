# /chabudai — 確信度への揺さぶり（ちゃぶ台返し）

Wikiの記述に**知的な揺さぶり**をかけ、確信度がどこで水増し・確証バイアス・反証の見落としに支えられているかを暴き、
**根拠づけて確信度を引き下げる**。狙いは、下がった数値が生む認知不協和から自分のバイアスに気づき、
**新しい探索域（新規仮説・再検証）を開く**こと。既存グローバルスキル `yusaburi`（厳格な思考メンター）の哲学を、
Wikiレコードに規律をもって適用する版である。

引き下げは**根拠づけてのみ**行う。揺さぶり材料を新規sourceに残し、`self-reflection` 活動レコード（ACT）に紐づけて
**提案 → ユーザー承認 → 反映**する。挑発のために数値を無根拠に動かすことはしない（下げるのも上げると同じ規律が要る）。

> **パス・ID規約**: 以下の `sources/` `wiki/` は現在のプロジェクト `projects/<slug>/` 配下を指す（現在のプロジェクトは `projects/current.md` の `current-project`）。ID とファイル名は**プロジェクト接頭辞つき**（例 `SELF-H-001`・`SELF-ACT-001`）。採番は種別×プロジェクトごとの既存最大+1。

## 手順

1. **現在プロジェクト・ステージを解決する** — `projects/current.md` の `current-project: <slug>` を読み、「プロジェクト一覧」表で接頭辞（PREFIX）を確定する。以降 `sources/` `wiki/` は `projects/<slug>/` 配下。`wiki/stage.md` で現在ステージを確認し、`playbooks/<stage>.md` の重点仮説タイプと、`playbooks/interviewing.md` の**確証バイアス／反証型・発見型**の心得を読む（揺さぶりの拠り所）。

2. **スコープを決める（`AskUserQuestion`）** — 何を揺さぶるか確認する:
   - **①狙い撃ち**: ユーザーが指定した特定仮説（`H-NNN`）だけを深く揺さぶる。
   - **②全体スイープ**: `wiki/index.md` と `wiki/hypotheses/` を走査し、**確信度が高く重要度も高い仮説を優先**して揺さぶる（＝根拠が甘いまま数字だけ高い「危険象限」）。`importance: auto` は `CLAUDE.md` のステージ→重点タイプ表で解決する。
   - どちらでも、対象ごとに **仮説本文＋確信度履歴テーブル＋紐づく `[[ACT-NNN]]`＋参照している `sources/`** を必ず読んでから揺さぶる（記述の実体を見ずに批判しない）。

3. **揺さぶる（批判的監査）** — グローバル `yusaburi` の哲学（**盲点の暴露・誠実な辛口・代案なき批判をしない**）と `CLAUDE.md` の証拠の階梯で、対象ごとに次の観点を回す。該当したものを具体的な引用つきで指摘する:
   - **証拠の階梯不足** — 確信度7-8なのに根拠列が〈発言〉止まり、5-6なのに〈自認〉未満。数字が証拠の強さに追いついていないか（interest ≠ intent）。
   - **架空データ水増し** — 〈架空〉由来なのに確信度が上限8を超えている／「実データ未検証」なのにステータスが`検証済み`になっていないか。
   - **確証バイアス** — 支持証拠ばかり集め、反証を軽く扱っていないか。「この仮説が間違っている証拠を3つ挙げられるか」「反対派の最強の論拠は何か」を自問する。
   - **反証の未反映** — `log.md`・ACTの学習カードに反証・想定外があるのに、確信度・ステータスに反映されていないか。
   - **反証可能性の欠如** — 仮説文が「重要だ」「〜と思う」式の検証不能な言い切りになっていないか。何が観測されれば反証かが曖昧でないか。
   - **前提の脆さ** — 前提節の暗黙前提のうち、崩れたら仮説ごと崩れるものはどれか。それは検証されているか。
   - **サンプルの弱さ** — 単一出典・n=1・単一ラウンド・偏ったセグメントに依存していないか。
   - **別解釈（逆張り）** — 同じ事実を説明する対立仮説はないか。今の解釈は最有力か、それとも都合が良いだけか。
   > バイアスの指摘は手段であって目的ではない。名前を言うだけで終えず、**なぜ危険か／どう回避するか**まで示す。自分（監査者）も「この仮説は甘い」と決めつける確証バイアスに陥っていないか点検する。

4. **揺さぶり材料を sources/ に記録する（新規・不変）** — `sources/YYYY-MM-DD-chabudai-<topic>.md` を新規作成する。冒頭に必ず次を明記する:
   > ⚠️ **揺さぶり（自己批判・反証監査）メモ。新規の実観測証拠ではなく、既存記録への批判的再解釈。これ自体は確信度を上げる根拠にならない。**

   対象仮説ごとに **危うい前提／バイアスの兆候／根拠不足の具体指摘（どの確信度行のどのタグが弱いか）／別解釈／反証仮説の種** を構造化して書く。**一度置いた `sources/` は以後編集しない**（不変ルール3）。既存のsourceファイルには一切手を触れない。

5. **self-reflection 活動レコード（ACT）を作る** — `wiki/activities/` の既存最大+1で `<PREFIX>-ACT-NNN` を採番し、`templates/activity.md` に従い `type: self-reflection` で作成する（`id` はファイル名と同じ接頭辞つき。`.gitkeep` が残っていれば削除してよい）。
   - 本文冒頭に `対象仮説: [[<PREFIX>-H-NNN]] …`（wikilink）と `揺さぶり材料: sources/YYYY-MM-DD-chabudai-<topic>.md`（相対mdリンク）を書く。frontmatter `hypotheses` は接頭辞つき配列。
   - **テストカード**（事前登録・後から書き換えない）: 目的＝各仮説の確信度の**根拠を反証しに行く**／方法＝証拠の階梯・バイアス観点での批判的再監査／指標＝根拠列のタグが現在の確信度を支えているか／**成功基準＝現在の確信度が証拠の階梯・不変ルールで正当化できなければ引き下げ、正当化できれば据え置き（＝反証耐性の確認）**。
   - **学習カード**（監査後に記入）: 事実（記録に実際に何が書かれているかを引用）／解釈（どこが根拠不足か・どのバイアスか、事実と分けて）／驚き・想定外／確信度の更新テーブル／次のアクション。

6. **確信度の引き下げを提案する（提案 → 承認 → 反映）** — 対象ごとに **現在値 → 提案値・提案ステータス・理由（発火した観点）** を提示する。**必ずユーザーの承認を得てから反映する**。
   - **引き下げは、現在値が証拠の階梯・不変ルールで正当化できないと示せる場合に限る**（例: 7だが根拠が〈発言〉のみ→5、〈架空〉由来で9→8、反証が出ているのに据え置き→引き下げ＋ステータス`反証`）。
   - 揺さぶっても崩れなければ**据え置き**とし、その旨を1行記録する（＝反証耐性の確認）。
   - **このスキルは確信度を下げる／据え置くのみ。机上監査で確信度を上げない**（9-10は実観測が反証を耐えた場合に限るため。上げ直しは実証拠を得て `/ingest` で行う）。
   - 承認後、仮説レコード `wiki/hypotheses/<PREFIX>-H-NNN.md` の frontmatter `confidence`/`status` を更新し、確信度履歴テーブルに1行追記する（活動列に `[[<PREFIX>-ACT-NNN]]`、根拠列の先頭に証拠種別タグ＋「根拠不足のため減点」等の理由）。ACT学習カードの確信度更新テーブルも埋める。

7. **新しい探索域を開く（認知不協和 → 次の一手）** — 揺さぶりで露出した**別解釈・反証仮説・見落とされた顧客/状況/前提**を、ACTの「次のアクション」に **新規仮説の種（`/formulate` へ）** と **再検証のテストカードの種（`/plan` へ）** として積む。ここが揺さぶりの payoff（下がった数字を眺めて終わりにしない）。有望なら新規 `H-NNN` の起票を提案し、承認後に `/formulate` 相当で作成してこのACTに紐づける（事後発見は事前登録した検証ではないため確信度を上げすぎない）。

8. **index と log を更新する** — `wiki/index.md` の該当行を更新する。`wiki/log.md` に**追記のみ**で1行足す:
   `## [YYYY-MM-DD] self-reflection | <PREFIX>-ACT-NNN 揺さぶり監査 → <PREFIX>-H-NNN 確信度X→Y（根拠不足）／<PREFIX>-H-MMM 据え置き（反証耐性確認）`

## 守ること

- 揺さぶりは**記録への批判であって、人格や過去の意思決定者への攻撃ではない**。欠陥を突くなら必ず**別解釈か次の検証**を添える（グローバル `yusaburi` の「代案なき批判をしない」を継承）。
- 確信度を**下げるのも上げると同じく規律が要る**。根拠なく数値を動かさない。引き下げは現在値が証拠の階梯・不変ルールで正当化できないと示せる場合に限り、挑発目的で数値操作しない。
- **机上監査で確信度を上げない**（下げる／据え置くのみ）。9-10 は実観測の反証耐性に限る。
- `sources/` は**新規追加のみ・不変**。既存sourceを改変・削除しない。揺さぶりメモ冒頭に「新規証拠ではない」旨を必ず明記し、実証拠と誤認させない。
- テストカードの成功基準は開始後に書き換えない（後知恵バイアス防止・不変ルール6）。`log.md` は追記のみ（過去行の編集禁止）。
- 確信度・ステータスの反映は**必ず承認を得てから**。非対話・一括実行でも、機械的に〈反証〉が確定し引き下げが証拠の階梯の範囲に収まる場合に限り提案内容を明示のうえ反映してよいが、解釈を要する引き下げは対話で承認を得る。
