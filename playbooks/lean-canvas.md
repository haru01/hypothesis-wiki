# リーンキャンバスの心得（本家 Running Lean 準拠）

事業モデルを1枚で捉える**リーンキャンバス**（Ash Maurya『Running Lean』）の方法論・思想。
`/lean-canvas` スキルが参照する**心得（Why）**の正典。現場のグリル手順（How）は
[../.claude/skills/lean-canvas/SKILL.md](../.claude/skills/lean-canvas/SKILL.md) が持つ
（心得＝Why はここ、進行文言＝How はスキル。`cpf.md` ↔ `interviewing.md` と同じ分担）。

ブロックと本リポジトリの仮説タイプの**対応（写像）は [../ontology.md](../ontology.md) の
「リーンキャンバス（仮説検証への写像）」が正本**（ここには再掲しない＝二重管理・ドリフト防止。
正本の正本は `ontology.yaml` の `lean-canvas`）。

## 位置づけ — キャンバスは「検証すべき前提の束」

リーンキャンバスは**Plan A のスナップショット**であって、実行計画書ではない。埋まった9ブロックは
**まだ検証されていない前提（仮説）の集まり**であり、キャンバスの価値は「守る」ことでなく**「試す」**ことにある。
本家の精神は "Life's too short to build something nobody wants"（誰も欲しがらないものを作るには人生は短すぎる）。
だから埋めた達成感に酔わず、**どこが最も脆いか**を探すために使う。

本リポジトリでは、各ブロックのエントリを**証拠の階梯**（[../ontology.md](../ontology.md) の evidence-ladder）で
`検証済み`/`未検証` に判定する。埋まっただけのキャンバスは **仮説キャンバス**であり、検証済みではない（paper fit ≠ fit）。

## 9ブロックの主眼（本家準拠）

| ブロック | 本家の主眼 |
|---|---|
| 顧客セグメント + アーリーアダプター | 誰のためか。**アーリーアダプター**＝いま最も強く痛みを感じ、代替手段にコストを払ってでも解こうとしている先鋒 |
| 課題 + 既存の代替手段 | 上位1〜3の課題と、顧客が**今それをどう凌いでいるか**（既存の代替手段＝真の競合。後述） |
| 独自の価値提案 (UVP) | なぜ違い・注目に値するかを**一文**で。機能でなく成果。**ハイレベルコンセプト**＝「（既知のX）の（Y）」比喩 |
| ソリューション | 上位課題を解く**最小**の機能。課題と対で紐づく（課題のないソリューションは持たない） |
| チャネル | 顧客への到達経路。**ステージで意味が変わる**（後述の stage-lens） |
| 収益の流れ | 課金モデル・単価・支払い意思(WTP)。**価格も仮説** |
| コスト構造 | 主要な変動費・固定費（人・インフラ・顧客獲得コスト） |
| 主要指標 | 事業が効いているかを測る**少数**の行動指標（顧客工場＝AARRR） |
| 圧倒的優位性 | 容易に**買えず・真似できない**もの。**最後に"発見"する**（最初に埋まらなくてよい） |

## 記入順 vs 検証順

- **記入順（sketch order）**＝網羅のための順: 顧客セグメント → 課題 → UVP → ソリューション → チャネル →
  収益＋コスト → 主要指標 → **圧倒的優位性（最後）**。優位性を最後に置くのは「最初に分かるものでなく、
  検証の過程で発見されるもの」だから。
- **検証順（validation order）＝ `riskiest-first`**: 本家は明確に「**左→右・上→下で埋めない。確信度の低い＝
  最もリスキーな前提から検証する**」と言う。記入で全ブロックを俯瞰したら、次は**最重要×証拠なし**の前提を選び、
  そこから壊しにいく。本リポジトリでは ACT frontmatter の `riskiest-assumption` と `/plan` の重要度×証拠マップで
  優先度を決める（アサンプションマッピングの「重要×証拠なし」象限）。

## ブロックの意味はステージで変わる（stage-lens）

本家の3段階（課題解決フィット → PMF → スケール）でブロックのレンズが変わる。正本の表は
[../ontology.md](../ontology.md) の stage-lens。要点:

- **チャネル** — 初期は "**Build a Path to Customers from Day 1**"。**スケールする流通**である前に、まず
  **想定顧客に会って学ぶための経路**（インタビュー・デモに到達する手段）。本リポジトリのステージで言えば
  CPF/PSF では「会える経路があるか」を問い、スケール流通の評価は PMF 以降。**チャネルを"スケール"だけで
  詰めると、初期に最も重要な"顧客に到達して検証できるか"を落とす。**
- **主要指標** — 初期は顧客工場(AARRR: 獲得→活性→維持→収益→紹介)の**どの一歩が詰まっているか**を見る学習指標。
  後期に事業の健全性指標。虚栄指標（PV・登録数など行動につながらない数字）を避ける。
- **収益の流れ** — **価格も仮説**。初期は〈支払い〉意思（対価・前払い・導入コミット）の検証対象であり、
  「払うと思う」という発言は証拠にしない（interest ≠ intent）。

## Customers（払う人） vs Users（使う人）

本家は両者の区別を明示する。**払う人と使う人が別**なら（例: 決裁するマネージャ vs 実際に使う担当者）、
それは別セグメントであり、**1枚のキャンバスは片方の主役に固定する**（両立させると誰にも刺さらない）。
初期(CPF)は「誰が払うか」より**「誰の痛みが本物か」**で主役を選ぶのが定石（収益・支払いの検証は PSF 以降）。
もう片方は別キャンバスに回す。

## 既存の代替手段 ＝ 真の競合

顧客は今も何かで凌いでいる（他社製品・自作・我慢・**何もしない=非消費**）。この**既存の代替手段こそが真の競合**。
代替手段が一つも挙がらない課題は「本当に痛いのか」を疑う（誰も解こうとしていない痛みは痛みでない疑い）。

## 圧倒的優位性の例（買えず・真似できない）

多くのマスは*本物の*優位性ではない。**機能・先行者・情熱は優位性でない**（金や時間で手に入る）。本家が挙げる例:
インサイダー情報／ドリームチーム／個人の権威・ブランド／コミュニティ／既存顧客・実績／独自のSEO・ネットワーク効果。
空欄が正直なら空欄でよい（検証の過程で発見する）。

## よくある誤用

- **セグメント混在** — 複数顧客を1枚に混ぜる（→ 1枚1セグメントに固定）。
- **課題理解前にソリューション記入** — 顧客・課題が固まる前に解決へ飛ぶ（CPF規律違反）。
- **UVPが機能列挙・差別化なし** — 成果でなく機能、既存代替との違いが言えない。
- **優位性が実は機能・先行者** — 買えるもの・真似できるものを優位性と誤認。
- **収益・コストにWTP証拠ゼロ** — 「払うと思う」で確信する（interest ≠ intent）。
- **虚栄指標** — 行動につながらない見栄え指標。
- **埋めた＝検証済みの誤認** — 埋まったキャンバスを検証済みと錯覚する（paper fit ≠ fit）。
